1. 项目概述

1.1 要解决什么问题

将 Shopify 在售商品的 SKU 与公司货源商品 SKU 进行智能对齐。核心痛点：

  • 跨语言：Shopify 英文描述 vs 货源中文描述
  • 跨粒度："黑色XXL"（拆分）vs "黑色大码"（合成）
  • 跨语义："6-15岁"（精确）vs "儿童装"（模糊）
  • 跨基数：目标 5 个 SKU，货源 3 个 SKU（缺货）
  • 跨组合：套装(上衣+短裤) vs 货源拆分售卖
  • 跨数量："3件装" vs "单件"

1.2 设计目标

场景自动化目标人工介入
完全一致100% 自动无
语言不同AI 转换后自动无
名称不同AI 理解后自动无
规格表达不同AI 归一化后自动低置信度时确认
SKU 数量不同AI 建立关系审核确认
货源不足AI 自动找货审核找货结果
无法判断—人工处理

1.3 核心设计原则

  1. 不直接匹配 SKU 文本，先建立商品知识层——所有商品先解构为结构化实体，再做属性对齐
  2. 原子化解构——把 SKU 拆到最底层组件 + 数量，单件是组件数=1 的特例
  3. 三阶段漏斗匹配——粗筛→属性匹配→LLM 推理，逐层精细化控制成本
  4. 置信度三分流——自动执行 / 人工确认 / 自动找货，阈值按品类动态调整
  5. 知识持久化——商品实体、匹配结果、人工修正全部存储，越用越准
  6. AI 修复中间层——匹配失败不直接丢人工，先 AI 自动重试（翻译/同义词/图搜）

2. 系统架构

系统分为 6 层，每层取自融合方案的最强来源：

1

商品知识图谱层 (Product Knowledge Graph)

所有商品入库即创建 Entity：结构化属性 + 文本 embedding + 图片 embedding。持久化存储，支持增量更新和跨商品知识复用。

来源: 方案 C
2

原子化解构层 (Atomic Deconstruction)

SKU → [{item, attributes, quantity}, ...]。单件是长度 1 的特例，套装自然拆解。在此基础上应用属性等价映射表。

来源: 方案 B
3

三阶段匹配漏斗 (Matching Funnel)

粗筛（商品级向量）→ 属性匹配（组件级规则+映射表）→ LLM 推理（灰区判定）。文本+图片双 embedding。

来源: 方案 A + C
4

关系推理 + 置信度路由 (Relation & Routing)

关系类型判断（1:1 / 1:N / N:1 / N:M）→ 置信度评分（动态阈值）→ 路由（自动 / AI修复→人工 / 自动找货）。

来源: A + B + C
5

三模式审核台 (Review Console)

批量模式（高量1:1）/ 积木模式（套装1:N）/ AI 修复模式（批量异常重试）+ 仪表盘概览。

来源: A + B + C
6

学习闭环 (Learning Loop)

知识图谱积累 + 模板记忆（套装公式参数化复用）+ 模型微调（定期 fine-tune）。

来源: A + B + C

3. SKU 关系矩阵

这是整个系统的核心认知模型。SKU 之间存在 4 种关系类型，必须全部支持：

关系类型场景示例说明DB 映射表
1:1 Shopify "Black XXL" → 货源 "黑色加大码" 最简单，一个目标 SKU 对应一个货源 SKU sku_mapping_1to1
1:N Shopify "套装(上衣+短裤)" → 货源A上衣 + 货源B短裤 一个目标 SKU 拆成多个货源 SKU sku_mapping_1toN
N:1 Shopify 5个SKU(黑S/M/L + 白S/M) → 货源2个(黑/白) 多个目标 SKU 合并到一个货源 SKU（粒度不同） sku_mapping_Nto1
N:M 3件装套装(2上衣+1短裤) → 货源A(上衣) + 货源B(短裤) 最复杂，多对多（1:N + 数量倍数的组合） sku_mapping_1toN + quantity
⚠️ 数据库设计要点
传统数据库大多只设计了 1:1 映射。本系统必须支持 1:N 和 N:1，需要改写 DB Schema。N:M 本质上是 1:N + 数量系数的组合，不需要单独的映射表。

系统由 5 个 Agent 协作完成，每个 Agent 职责单一，通过事件驱动编排：

5.2 Agent 1: 商品理解 Agent (Product Understanding)

职责

判断 Shopify 商品和货源商品是否是"同一个东西"（商品级粗筛）。这是匹配漏斗的第一步。

输入

{
  "shopify_product": {
    "title": "Women's Summer Set (Top + Shorts)",
    "description": "...",
    "images": ["url1", "url2"],
    "category_hint": "clothing"
  },
  "supplier_product": {
    "title": "女士夏季两件套 上衣+短裤",
    "description": "...",
    "images": ["url1", "url2"]
  }
}

LLM Prompt 模板

你是一个电商商品对齐专家。判断以下两个商品是否是同一个商品（或同一款的不同版本）。

【Shopify 商品】
标题: {shopify_title}
描述: {shopify_description}
图片描述: {shopify_image_captions}

【货源商品】
标题: {supplier_title}
描述: {supplier_description}
图片描述: {supplier_image_captions}

请输出 JSON:
{
  "is_same_product": true/false,
  "similarity_score": 0.0-1.0,
  "reasons": ["图片一致", "类目一致", "颜色一致"],
  "differences": ["尺码体系不同"],
  "category_confirmed": "kids_jacket"
}

输出

{
  "is_same_product": true,
  "similarity_score": 0.92,
  "reasons": ["图片一致", "类目一致", "颜色一致", "风格一致"],
  "differences": ["尺码体系不同"],
  "category_confirmed": "summer_set"
}

关键决策

  • similarity_score ≥ 0.7 → 进入 SKU 级匹配流程
  • similarity_score < 0.7 → 标记为"商品不匹配"，跳过该商品对
  • 使用 text-embedding-3-small 做文本向量粗筛（成本极低），只有向量相似度 > 0.75 的才调用 LLM
  • 图片 embedding 用 CLIP 模型预计算，作为辅助信号

5.4 Agent 3: 关系推理 Agent (Relation Reasoning)

职责

在组件级别建立 SKU 之间的映射关系，判断关系类型（1:1 / 1:N / N:1），生成映射方案。

处理逻辑

// 伪代码
function reasonRelations(shopify_skus, supplier_skus):
    # 1. 对每个 Shopify SKU 的每个组件，找最佳匹配的货源组件
    for shopify_sku in shopify_skus:
        for component in shopify_sku.components:
            best_match = findBestComponentMatch(
                component,
                supplier_skus.flatMap(s => s.components)
            )

    # 2. 判断关系类型
    if shopify_sku.component_count == 1 and best_match.sku.component_count == 1:
        relation = "1:1"

    elif shopify_sku.component_count > 1:
        # Shopify 是套装，需要多个货源
        relation = "1:N"
        # 检查是否所有组件都找到了货源
        missing = components_without_match
        if missing.length > 0:
            trigger_auto_sourcing(missing)

    elif supplier_sku_mapped_to_multiple_shopify:
        # 多个 Shopify SKU 指向同一个货源 SKU
        relation = "N:1"
        # 记录缺失属性（如货源没有尺码信息）
        record_missing_attributes()

    # 3. 计算组合置信度
    if relation == "1:N":
        # 套装整体置信度 = 各组件置信度的加权平均
        bundle_score = weightedAvg(component_scores)
    elif relation == "N:1":
        # 粒度合并置信度 = 各 Shopify SKU 置信度的最小值（短板效应）
        merge_score = min(sku_scores)

    return { relation, mapping_plan, confidence }

关系判断规则

条件关系类型处理
Shopify 1组件 → 货源 1组件1:1直接映射
Shopify 多组件 → 各自找到货源1:N多货源映射
Shopify 多组件 → 部分缺失1:N (部分)触发自动找货
Shopify 多SKU → 同一货源SKUN:1记录缺失属性
Shopify 单组件 quantity>1 → 货源 quantity=11:1 + 倍数设 quantity_multiplier

6. 匹配引擎与评分

6.1 三阶段漏斗设计

Stage 1: 商品级粗筛 (Coarse Filter)

项目说明
目标把 N×M 笛卡尔积缩减到可管理规模
方法文本 embedding + 图片 embedding 双向量余弦相似度
模型text-embedding-3-small (文本) + CLIP (图片)
阈值文本相似度 > 0.75 或 图片相似度 > 0.80
成本极低（预计算向量，纯数据库查询）
产出候选商品对列表
-- pgvector 查询示例
SELECT s.id as shopify_id, sup.id as supplier_id,
       s.text_embedding <=> sup.text_embedding as text_sim,
       -- 图片取最相似的一对
       max_image_similarity(s.image_embeddings, sup.image_embeddings) as img_sim
FROM product_entities s
CROSS JOIN product_entities sup
WHERE s.source = 'shopify' AND sup.source != 'shopify'
  AND (s.text_embedding <=> sup.text_embedding) < 0.25  -- cosine distance < 0.25
  AND s.category = sup.category  -- 类目必须一致
ORDER BY text_sim ASC
LIMIT 50;  -- 每个shopify商品最多50个候选

Stage 2: 组件级属性匹配 (Attribute Matching)

对候选商品对的每个 SKU 的每个组件，逐属性对比：

匹配类型示例分数说明
精确匹配black = black1.0标准化后完全相同
等价匹配大码 = XXL0.9映射表命中
模糊匹配深蓝 ≈ navy0.7-0.85向量相似度
LLM 推断儿童装 ≈ age 6-150.6-0.9LLM 判断
缺失属性一方无材质不扣分但降权权重 × 0.8
冲突属性black vs white0.0直接否决

组件匹配评分公式

// 单组件匹配分数
component_score = Σ(attribute_score_i × weight_i) / Σ(weight_i)

// 属性权重（可按品类调整）
weights = {
  "color":    0.30,   // 颜色最重要
  "size":     0.25,   // 尺码
  "material": 0.15,
  "pattern":  0.10,
  "gender":   0.10,
  "style":    0.05,
  "season":   0.05
}

// 如果某属性一方缺失，该属性不参与计算，但总权重降低
// 如果某属性冲突（如颜色不同），component_score 直接 = 0

// 套装整体分数 (1:N)
bundle_score = Σ(component_score_i × component_weight_i) / Σ(component_weight_i)
// component_weight 默认均等，可配置

// N:1 粒度合并分数
merge_score = min(all_sku_scores)  // 短板效应
// 因为如果有一个 SKU 匹配不好，整个合并就不可靠

Stage 3: LLM 灰区推理 (LLM Reasoning)

对 Stage 2 分数在 0.5-0.9 之间的组件对，调用 LLM 做最终判断：

你是 SKU 匹配专家。判断以下两个 SKU 组件是否等价。

【Shopify 组件】
原始描述: "{shopify_raw}"
标准化属性: {shopify_attributes}

【货源组件】
原始描述: "{supplier_raw}"
标准化属性: {supplier_attributes}

请判断:
1. 这两个组件是否可以视为同一商品？
2. 如果不等价，差异在哪里？
3. 置信度是多少 (0.0-1.0)？

输出 JSON:
{
  "is_equivalent": true/false,
  "confidence": 0.0-1.0,
  "reason": "...",
  "risk_factors": ["..."]
}

6.2 匹配漏斗成本控制

成本控制策略
  • Stage 1：纯向量查询，成本极低，过滤掉 90%+ 的不相关商品对
  • Stage 2：规则 + 映射表，成本极低，处理大部分明确匹配
  • Stage 3：LLM 调用，成本较高，只处理灰区（约 10-20% 的 SKU 对）
  • 属性提取缓存：相同 SKU 描述不重复调用 LLM
  • 模型分层：简单 SKU 用 GPT-4o-mini，复杂 SKU 才升级到 GPT-4o

7. 置信度与路由

7.1 置信度评分体系

分数区间路由人工介入示例
≥ 0.90 自动匹配，直接绑定 无 Black XXL = 黑色XXL（标准化后完全一致）
0.70 ~ 0.90 AI 修复模式 → 成功则自动 / 失败则人工审核 有，但 AI 给推荐 6-15 Years ≈ 儿童款（需 AI 推理）
0.50 ~ 0.70 人工审核台 有，需人工确认 女士运动鞋 vs 女鞋（类目模糊）
< 0.50 触发自动找货 仅审核找货结果 找不到匹配的货源 SKU

7.2 动态阈值调整

阈值不是固定的，按以下维度动态调整：

// 伪代码：动态阈值计算
function getThresholds(category, history):
    # 基础阈值
    base = { auto: 0.90, repair: 0.70, manual: 0.50 }

    # 按品类调整
    if category in STANDARDIZED_CATEGORIES:  # 电子配件、标品
        base.auto -= 0.05   # 放宽自动匹配
    elif category in CUSTOM_CATEGORIES:      # 服装、个性化
        base.auto += 0.03   # 收紧自动匹配

    # 按历史准确率调整
    accuracy = getHistoricalAccuracy(category)
    if accuracy > 0.95:
        base.auto -= 0.03   # 历史表现好，放宽
    elif accuracy < 0.85:
        base.auto += 0.05   # 历史表现差，收紧

    # 按反馈数据量调整（冷启动期保守）
    feedback_count = getFeedbackCount(category)
    if feedback_count < 100:
        base.auto = 0.95    # 冷启动期非常保守

    return base

7.3 路由决策流程

match_score = computeScore(component_matches)

thresholds = getThresholds(category, history)

if match_score >= thresholds.auto:
    → auto_match()           # 自动绑定，记录到映射表

elif match_score >= thresholds.repair:
    → ai_repair()            # AI 修复模式
      → if repair_success: auto_match()
      → else: send_to_review()

elif match_score >= thresholds.manual:
    → send_to_review()       # 人工审核台

else:
    → auto_source()          # 触发自动找货
    → send_to_review()       # 找货结果需人工审核

8. 自动找货

8.1 缺口检测

匹配完成后，统计未匹配的 Shopify SKU 组件，生成「缺口属性画像」：

// 缺口检测伪代码
function detectGaps(shopify_product, matched_mappings):
    all_components = shopify_product.skus.flatMap(sku => sku.components)
    matched_components = matched_mappings.map(m => m.component)

    gaps = all_components.filter(c => !matched_components.includes(c))

    return gaps.map(gap => ({
        missing_component: gap.item_name,
        missing_attributes: gap.attributes,
        search_keywords: generateSearchKeywords(gap),
        // 生成搜索关键词: "黑色 女式短裤 L码"
    }))

function generateSearchKeywords(component):
    keywords = []
    if component.attributes.color: keywords.push(component.attributes.color)
    if component.attributes.size: keywords.push(component.attributes.size.value)
    keywords.push(component.item_name)
    // 加上翻译后的关键词
    keywords.push(translate(component.item_name, 'zh'))
    return keywords

关键注意事项

⚠️ 必须注意
  1. 不要纯靠 LLM 做匹配——LLM 只做属性提取和灰区推理，匹配逻辑必须代码化。纯 LLM 匹配不可控、不可解释、成本高。
  2. 属性提取要做缓存——相同 SKU 描述不重复调用 LLM，这个优化能节省 50%+ 的 LLM 成本。
  3. 冷启动期阈值要保守——前期 auto 阈值设 0.95+，宁可多走人工，也不要错配。随着反馈数据积累再放宽。
  4. 映射表需要人工审核机制——AI 建议的新映射不能自动生效，必须人工确认后才加入。
  5. 1:N 映射的 DB Schema 要提前设计——即使 V1 不实现套装拆解，表结构也要预留好，避免后期大改。
  6. 图片 embedding 预计算——商品入库时就计算好 embedding，不要在匹配时实时计算。
